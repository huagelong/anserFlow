# Docs Wiki Builder

## 使用目标

把项目根目录的 `docs/*.md` 整理成 `wiki/` 下的知识库，方便 Codex 或其他大模型按总目录逐级检索。

必须遵守：

- 只从 `docs/*.md` 获取内容。
- 不修改 `docs` 原文。
- 不改动代码、配置和无关目录。
- 不补写 `docs` 中没有的设计结论。
- 不创建临时文件或额外总结文档。

## 输出结构

在项目根目录生成或更新：

```text
wiki/
  README.md
  <源文档名>/
    README.md
    <章节>.md
    <章节>/
      README.md
      <子章节>.md
```

如果本 Skill 的副本也放在项目 `wiki/flowcode_wiki/` 下，生成业务 wiki 时不要覆盖或移动这个 Skill 目录。

## 拆分规则

按源文档文件名顺序处理 `docs/*.md`。

解析标题时必须忽略代码块中的 `#`、`##`、`###`。推荐粒度：

- 每个源文档的一级标题成为 `wiki/<源文档名>/README.md` 的主题入口。
- 二级标题没有三级标题时，二级章节拆成一个小文档。
- 二级标题下面有三级标题时，二级章节生成一个目录入口，三级标题各自拆成小文档。
- 不要把表格、代码块、流程图、连续枚举步骤拆断。
- 章节正文尽量原样迁移，只允许添加导航元信息。

## 总目录要求

`wiki/README.md` 必须包含：

- 快速入口：所有源文档主题入口。
- 推荐检索路径：面向常见问题的阅读路线。
- 关系图谱：Mermaid 图或关系表。
- 细分文档树：所有小文档链接。

推荐检索路径可以从这些链路开始：

- 产品全貌：项目概述 -> 系统架构 -> 数据模型 -> API 设计
- 需求到代码闭环：项目概述 -> 工作流引擎 -> AI 编程工具集成 -> Git 集成方案 -> 测试策略
- 插件与外部工具：插件与钩子体系 -> AI 编程工具集成 -> Git 集成方案 -> API 设计
- 线上反馈到 Issue：监控与反馈闭环 -> 工作流引擎 -> 数据模型 -> API 设计
- 部署与验证：部署与运维 -> 监控与反馈闭环 -> 测试策略

## 关系规则

优先使用源文档编号和主题建立关联：

- `01-项目概述` 关联 `02-系统架构`、`03-数据模型`、`04-API设计`
- `02-系统架构` 关联 `03-数据模型`、`04-API设计`、`05-工作流引擎`、`10-部署与运维`
- `03-数据模型` 关联 `04-API设计`、`05-工作流引擎`、`09-监控与反馈闭环`
- `04-API设计` 关联 `03-数据模型`、`05-工作流引擎`、`06-插件与钩子体系`、`11-测试策略`
- `05-工作流引擎` 关联 `06-插件与钩子体系`、`07-AI编程工具集成`、`08-Git集成方案`、`09-监控与反馈闭环`、`11-测试策略`
- `06-插件与钩子体系` 关联 `07-AI编程工具集成`、`08-Git集成方案`、`04-API设计`
- `07-AI编程工具集成` 关联 `05-工作流引擎`、`08-Git集成方案`、`10-部署与运维`、`11-测试策略`
- `08-Git集成方案` 关联 `05-工作流引擎`、`07-AI编程工具集成`、`10-部署与运维`

- `11-测试策略` 关联 `04-API设计`、`05-工作流引擎`、`07-AI编程工具集成`、`10-部署与运维`

如果实际 `docs` 文件名不同，按标题语义匹配，不要创造不存在的主题。

## 小文档导航

每个小文档开头添加简短导航块：

```md
> 来源：`docs/xx-xxx.md` 第 N 行
> 位置：[总目录](../README.md) -> [源文档](README.md) -> 当前章节
> 相邻：[上一篇](...) · [下一篇](...)
> 相关主题：[...](...) · [...]
```

如果用户要求完全保留原文，不要在小文档内加入导航块，把所有关系集中放到 `wiki/README.md`。

## 校验

完成后检查：

- `git diff -- docs` 为空。
- `git diff --stat` 只出现 `wiki/`，或只出现用户明确允许的位置。
- `wiki/README.md` 能到达所有主题入口和小文档。
- 小文档相对链接能指向已生成文件。
- 代码块没有被标题解析误拆。

向用户汇报时只说明变更位置、拆分粒度和校验结果。
