**Issue 自动分析方案**（混合：关键词计分 + AI 异步补充分类）：

```
Issue 创建 → 关键词计分（同步，0ms）
                │
        ┌───────┴────────┐
        ▼                ▼
   高置信度(≥0.8)     低置信度 / 未命中
        │                │
        ▼                ▼
   直接采用分类      投递 Asynq → opencode 前置分析
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
   isBug 判定    needsCodeChange     分类
                        │               │
                        ▼               ▼
                 分析结果回写       分类结果异步回写
                        │
                        ▼
              审核通过后生成执行计划（planSummary / planDetail）
```

**opencode 前置分析**（任何 Issue 创建后均执行，在分类之前）：

| 分析项 | 说明 | 输出 |
|--------|------|------|
| isBug | 是否为缺陷/异常报告 | true / false |
| needsCodeChange | 是否涉及代码修改 | true / false |
| analysisReason | 分析推理过程 | 自然语言文本 |

> 前置分析在关键词计分后、AI 分类前运行。纯文档/配置类 Issue 可直接判定 `needsCodeChange=false`，跳过代码执行流程。
> 关键词计分离散度高的 Issue（例如标题同时命中 bug 和 feature）由 opencode 综合分析给出更准确的 isBug 判定。

**执行计划产物**（审批通过后、正式执行前生成）：

| 字段 | 说明 | 输出 |
|------|------|------|
| planSummary | 计划摘要 | 一段面向人阅读的短说明 |
| planDetail | 完整计划 | Markdown，含步骤、文件范围、验证策略 |
| planCompleted | 是否已具备可执行计划 | true / false |
| planGeneratedAt | 最近生成时间 | TIMESTAMP |

**关键词计分规则**（由 Classifier 模块执行）：

| 关键词 | 分类 | 权重 |
|--------|------|:---:|
| bug / 修复 / bugfix / 报错 / 异常 / 崩溃 / 错误 / error / crash | bug | 2.0 |
| 新增 / 添加 / 增加 / 实现 / 开发 / 功能 / feature / add | feature | 2.0 |
| 重构 / 优化 / 改进 / 整理 / 清理 / refactor / improve | refactor | 2.0 |
| 文档 / 说明 / 注释 / readme / docs / document | docs | 2.0 |
| 部署 / ci / cd / 环境 / 配置 / 基础设施 / deploy / infra / chore | infra | 2.0 |
| 测试 / 用例 / test / spec | test | 2.0 |
| 修复 / 改进 (弱匹配) | bug / refactor | 1.0 |
| 升级 / 迁移 / upgrade / migrate | infra | 1.0 |

> 英文关键词通过 strings.ToLower 统一处理。多关键词权重累加，最高分对应分类。
> 置信度 = 最高分 / (最高分 + 次高分)，≥0.8 直接采用，<0.8 触发 AI 异步补充分类。

### 3.2.4b Comment Issue 评论表

多用户可在 Issue 下协作讨论，支持 Markdown 内容和嵌套回复（楼中楼）。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) PK | UUID 主键 |
| issueId | VARCHAR(36) FK→Issue | 所属 Issue |
| authorId | VARCHAR(36) FK→User | 评论作者 |
| body | TEXT | 评论内容 (Markdown) |
| parentId | VARCHAR(36) FK→Comment | 父评论 ID（NULL = 顶层评论，非 NULL = 楼中楼回复） |
| editedAt | TIMESTAMP | 最后编辑时间（NULL = 未编辑过） |
| createdAt | TIMESTAMP | 创建时间 |
| updatedAt | TIMESTAMP | 更新时间 |

> 评论仅支持软删除（`deletedAt`），作者可编辑自己的评论。Issue 关闭后仍可继续评论。

```sql
-- Comment 索引
CREATE INDEX idx_comment_issue ON comments(issue_id, created_at);
CREATE INDEX idx_comment_parent ON comments(parent_id);
CREATE INDEX idx_comment_author ON comments(author_id);
```

|------|------|

| pending_review | 已提交，等待管理员审核 |

| failed | 测试运行失败 |

**result JSONB 字段结构**：

```go
// 存储到 TestCase.result (JSONB)
type TestResult struct {
    Status     string      `json:"status"`     // passed / failed
    DurationMs int64       `json:"durationMs"` // 执行耗时 (毫秒)
    Total      int         `json:"total"`      // 测试总数
    Passed     int         `json:"passed"`     // 通过数
    Failed     int         `json:"failed"`     // 失败数
    Skipped    int         `json:"skipped"`    // 跳过数
    Coverage   string      `json:"coverage"`   // 覆盖率 "82.5%"
    Output     string      `json:"output"`     // 测试原始输出 (截取前 10KB)
    Errors     []TestError `json:"errors"`     // 失败详情
    RunAt      time.Time   `json:"runAt"`      // 执行时间
}

type TestError struct {
    File    string `json:"file"`    // 失败文件
    Line    int    `json:"line"`    // 失败行号
    Message string `json:"message"` // 失败信息
}
```

**testScript JSONB 字段结构**（用户自定义测试脚本，优先级高于 framework 默认映射）：

```go
// 存储到 TestCase.testScript (JSONB)，为空时使用 framework 默认命令
type TestScript struct {
    Install string `json:"install,omitempty"` // 依赖安装命令: "go mod download"
    Command string `json:"command"`            // 测试执行命令: "go test ./pkg/... -v -race -cover 2>&1"
    Timeout int    `json:"timeout,omitempty"` // 超时秒数，默认 300
}
```

**变量替换规则**：

| 变量 | 替换为 | 示例 |
|------|--------|------|
| `{testFile}` | 测试代码在容器内的路径 | `/workspace/repo/pkg/auth/auth_test.go` |
| `{workspace}` | 容器工作目录 | `/workspace/repo` |
| `{projectDir}` | 项目根目录名 | `my-api` |

```json
// 示例 1：自定义 Go 子包测试 + race 检测
{
  "install": "cd {workspace} && go mod download",
  "command": "cd {workspace} && go test ./pkg/auth/... -v -race -cover 2>&1",
  "timeout": 120
}
```

```json
// 示例 2：前端 E2E 测试 (Playwright)
{
  "install": "cd {workspace} && npm install && npx playwright install",
  "command": "cd {workspace} && npx playwright test --reporter=list 2>&1",
  "timeout": 600
}
```

```json
// 示例 3：Java 多模块项目指定子模块
{
  "install": "cd {workspace} && mvn dependency:resolve -pl user-service -q",
  "command": "cd {workspace} && mvn test -pl user-service -B 2>&1",
  "timeout": 300
}
```

> **优先级**：`testScript.command` > `buildTestCmd(framework)`。若 testScript 未设置，回退到框架映射表。

### 3.2.5c HistoryRecord 历史记录表

所有核心实体 (Project / Requirement / Issue / TestCase) 的变更均自动记录到该表，支持时间线查看和版本对比。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) PK | UUID 主键 |
| targetType | VARCHAR(20) | 实体类型: project / requirement / issue / test_case |
| targetId | VARCHAR(36) | 实体 ID |
| action | VARCHAR(50) | 操作类型: created / updated / status_changed / deleted / reviewed |
| fieldName | VARCHAR(50) | 变更字段 (整体变更时为 "*") |
| oldValue | JSON | 变更前值 (NULL=新建) |
| newValue | JSON | 变更后值 |
| changedBy | VARCHAR(36) FK→User | 操作人 |
| createdAt | TIMESTAMP | 记录时间 |

**action 枚举说明**：

| action | 触发场景 |
|--------|---------|
| created | 实体创建 |
| updated | 字段修改 (title/description/priority 等) |
| status_changed | 状态流转 (draft→approved→running→done) |
| deleted | 实体软删除 |
| reviewed | 审核操作 (approve/reject) |
| test_generated | AI 生成测试代码完成 |
| test_executed | 测试运行完成 |
| rollback | 回滚到历史版本 |

**回滚快照策略**：

- 每次 `updated` / `status_changed` 操作自动记录 `oldValue`（变更前值）
- 回滚时：读取 `oldValue` → 写回实体 → 新建一条 `action=rollback` + `fieldName="*"` 记录，`oldValue` 为回滚前的值（可二次回滚撤销）
- 不存储完整快照，历史链本身就是可回溯的变更序列
