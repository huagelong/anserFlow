## 2.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| **前端** | Vue 3 + TypeScript | 渐进式框架，学习曲线平缓，适合小团队 |
| **构建工具** | Vite 5 | 极速 HMR，开发体验好 |
| **UI 组件库** | Element Plus | 成熟的企业级 Vue 3 组件库，中文生态好 |
| **状态管理** | Pinia | Vue 3 官方推荐，TypeScript 友好 |
| **路由** | Vue Router 4 | Vue 3 官方路由方案 |
| **后端框架** | Go + Gin | 高性能、低内存、编译型语言，部署简单 |
| **ORM** | Gorm v2 | Go 生态最成熟的 ORM，AutoMigrate 方便 |
| **数据库** | MySQL 8.0 | 广泛使用，JSON 类型支持，运维成熟 |
| **缓存/队列** | Redis 7 + Asynq | Go 原生任务队列（基于 Redis），轻量可靠 |
| **实时通信** | WebSocket (gorilla/websocket) | 双向通信，推送执行日志 |
| **文件存储** | 本地文件系统 | 沙盒产物、Skills 文件、日志归档 |
| **CLI 工具** | Go + Cobra | Go 生态标准 CLI 框架 |
| **容器化** | Docker + docker-compose | 一键部署 |
| **反向代理** | Nginx | 高性能反向代理，静态资源服务 |
| **CI/CD** | GitHub Flow + GitHub Actions | 标准分支策略 + 自动化构建 |
| **Git 操作** | Git CLI + go-git | git clone/push 常用 go-git（纯 Go 零依赖），ls-remote/rebase 降级 CLI |
| **Git 平台 SDK** | go-github + go-gitlab + go-gitea | PR/MR/Webhook 操作，三个平台均为必需依赖 |
| **状态机** | looplab/fsm | Go 生态最成熟的轻量 FSM，声明式事件+回调，Graphviz 可视化 |
| **沙盒容器** | Docker SDK | docker/client 官方包，容器内安全执行 AI 生成代码 |

**典型使用场景**：

```bash
# 1. 初始化项目（首次接入 flowcode）
flowcode init
# → ✓ Created .flowcode/config.yaml
# → ✓ Synced 7 skills to .flowcode/skills/
# → ✓ Committed & pushed to git

# 2. 查看项目概览
flowcode stats
# → Issues: 12 open / 5 in review / 3 blocked
# → Tests: 78% pass rate / 62% coverage
# → PRs: 2 open

# 3. 创建 Issue 并查看工作流状态
flowcode issue create -t "用户登录页" -d "实现邮箱+密码登录" -p p1
# → Created: iss_abc123
flowcode workflow status iss_abc123
# → State: draft | Next: submit / cancel

# 4. 运行测试
flowcode test run tc_xyz --watch
# → [实] 实时流式输出测试日志...

# 5. 查看历史并回滚
flowcode history list --issue iss_abc123
flowcode history rollback h_old_001 --confirm

# 6. PR 管理
flowcode pr list                          # 项目所有 PR
flowcode pr list --status open            # 只看打开的
flowcode pr create --issue iss_abc123 --branch feat/login
flowcode pr merge --issue iss_abc123       # 合并 PR

# 7. 升级 CLI
flowcode upgrade
# → Current: v1.2.3 | Latest: v1.3.0
# → Downloading... Done → Restart to apply
```

### 2.3.5 CLI 安全机制

CLI 工具直接操作云端的 Issue、触发执行、安装 Skill，安全性至关重要。

#### 2.3.5a Token 安全存储

```go
// internal/cli/auth/store.go

// 凭证按 OS 标准路径加密存储
// Linux:   ~/.config/flowcode/auth.json (权限 0600)
// macOS:   ~/Library/Application Support/flowcode/auth.json
// Windows: %APPDATA%/flowcode/auth.json

type CredentialStore struct {
    path string
    key  []byte // 从机器特征派生 (hostname + machine-id + os)
}

func (s *CredentialStore) Save(cred *Credentials) error {
    data, _ := json.Marshal(cred)
    // AES-256-GCM 加密后写入文件
    encrypted := aesGCMEncrypt(data, s.key)
    return os.WriteFile(s.path, encrypted, 0600)
}

func (s *CredentialStore) Load() (*Credentials, error) {
    encrypted, err := os.ReadFile(s.path)
    if err != nil {
        return nil, fmt.Errorf("not logged in, run 'flowcode login' first")
    }
    return aesGCMDecrypt(encrypted, s.key)
}

type Credentials struct {
    AccessToken  string    `json:"access_token"`
    RefreshToken string    `json:"refresh_token"`
    ServerURL    string    `json:"server_url"`
    ExpiresAt    time.Time `json:"expires_at"`
    DeviceID     string    `json:"device_id"`
}
```

```go
// internal/cli/auth/device.go

type DeviceInfo struct {
    DeviceID   string // SHA256(hostname + machine-id)
    DeviceName string // "MacBook Pro" / "CI Runner"
    OS         string // "darwin" / "linux" / "windows"
```

#### 2.3.5c 自动 Token 刷新

CLI 在请求前自动检查 Token 过期时间，透明刷新：

```go
// internal/cli/client/interceptor.go

func (c *Client) Do(req *http.Request) (*http.Response, error) {
    cred, _ := c.store.Load()

    // Token 将在 5 分钟内过期 → 提前刷新
    if time.Until(cred.ExpiresAt) < 5*time.Minute {
        newCred, err := c.refreshToken(cred.RefreshToken)
        if err != nil {
            return nil, fmt.Errorf("session expired, run 'flowcode login'")
        }
        c.store.Save(newCred)
        cred = newCred
    }

    req.Header.Set("Authorization", "Bearer "+cred.AccessToken)
    req.Header.Set("X-Device-ID", cred.DeviceID)
    return c.http.Do(req)
}
```

#### 2.3.5e CLI 安全最佳实践

| 措施 | 说明 |
|------|------|

### 2.3.6 WebSocket 分布式架构

#### 架构选型：Redis Pub/Sub + gorilla/websocket

flowcode 已依赖 Redis（Asynq 任务队列），复用为 WebSocket 消息总线，零额外组件：

```
┌──────────────────────────────────────────────────┐
│                    Redis                         │
│              Pub/Sub + Asynq                     │
└──────┬─────────────────────────┬────────────────┘
       │ channel: exec:{issueId} │
       │                         │
  ┌────▼─────┐            ┌──────▼─────┐
  │ Server A │            │  Server B  │
  │  ┌─────┐ │            │  ┌─────┐   │
  │  │ Hub │◄├─ subscribe ─┼──┤ Hub │   │
  │  └──┬──┘ │            │  └──┬──┘   │
  │     │    │            │     │      │
  │  ┌──┴──┐ │            │  ┌──┴──┐   │
  │  │ WS  │ │            │  │ WS  │   │
  │  │ WS  │ │            │  │ WS  │   │
  │  └─────┘ │            │  └─────┘   │
  └──────────┘            └────────────┘
```

**消息流**：
1. 用户连接 Server A 的 WebSocket（`/ws/executions/:issueId`）
2. Asynq Worker 在 Server B 执行完成 → 调用 `Hub.Publish(issueId, logLine)`
3. Server B Redis `PUBLISH exec:iss_abc "log line"`
4. 所有 Server 的 Hub 收到 → 转发给本地订阅该 `issueId` 的 WS 客户端

#### 代码结构

```go
// ws/hub.go — 每实例单例

type Hub struct {
    // 房间: issueId → 客户端集合
    rooms      map[string]map[*Client]bool
    register   chan *Client
    unregister chan *Client

    // Redis Pub/Sub
    rdb       *redis.Client
    pubsub    *redis.PubSub
    // 本地消息广播
    broadcast chan Message
}

func (h *Hub) Run() {
    // 订阅 Redis pattern: exec:*
    h.pubsub = h.rdb.PSubscribe(ctx, "exec:*")
    go h.listenRedis()

    for {
        select {
        case client := <-h.register:
            h.addToRoom(client)
        case client := <-h.unregister:
            h.removeFromRoom(client)
        case msg := <-h.broadcast:
            h.sendToRoom(msg.IssueID, msg.Data)
        }
    }
}

func (h *Hub) listenRedis() {
    for msg := range h.pubsub.Channel() {
        // exec:iss_abc → issueId = "iss_abc"
        issueId := strings.TrimPrefix(msg.Channel, "exec:")
        h.broadcast <- Message{IssueID: issueId, Data: []byte(msg.Payload)}
    }
}

// 发布消息到 Redis（任何实例调用）
func (h *Hub) Publish(issueId string, data []byte) {
    h.rdb.Publish(ctx, "exec:"+issueId, data)
}
```

```go
// ws/client.go — 每连接一个 goroutine 对

type Client struct {
    conn    *websocket.Conn
    issueId string
    send    chan []byte   // 写缓冲

}

    // 读客户端消息（心跳 pong），30s 无消息则断开
}

func (c *Client) writePump() {
    // 从 send channel 写入 WebSocket，30s 发一次 ping

```

```go
    IssueID string `json:"issueId"`

```

#### Nginx 分布式配置

多实例部署时，Nginx 通过 `ip_hash` 保证同一客户端始终连同一实例（维持 WebSocket 长连接），但消息通过 Redis 跨实例投递：

```nginx
upstream flowcode_server {
    ip_hash;                          # 会话保持
    server flowcode-1:3000;
    server flowcode-2:3000;
    server flowcode-3:3000;
}

server {
    location /ws/ {
        proxy_pass http://flowcode_server;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 3600s;     # 长连接超时
        proxy_send_timeout 3600s;
    }
}
```

> `ip_hash` 仅保证连接粘性；跨实例消息投递由 Redis Pub/Sub 保证。即使 `ip_hash` 失效导致客户端重连到另一实例，消息也不会丢失。

#### 技术对比

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| **Redis Pub/Sub** ✅ | 零额外组件、复用现有 Redis | 消息不持久化 | flowcode（日志流，丢失可接受） |
| Centrifugo | 功能全、自带鉴权/通道 | 额外部署维护 | 消息不可丢失场景 |
| NATS | 高性能、持久化 | 额外组件 | 高吞吐场景 |
| Kafka | 持久化、重放 | 过重 | 审计日志场景 |

**选型理由**：执行日志是实时流，断线后重连通过 `GET /api/v1/executions/:logId/stdout` 拉历史补全即可，无需消息持久化。Redis Pub/Sub 零成本满足需求。

### 2.3.7 开发工具链

#### 2.3.7a 热加载（air）

使用 [cosmtrek/air](https://github.com/cosmtrek/air) 实现代码变更自动重编译重启，避免每次修改后手动 `go build && ./flowcode server`：

```
开发工作流:
  保存 .go 文件
      │
      ▼
  air 检测文件变更
      │
      ▼
  自动 go build → 杀死旧进程 → 启动新进程
      │
      ▼
  浏览器刷新即可看到变更（< 3 秒）
```

`.air.toml` 配置：

```toml
# .air.toml
root = "."
tmp_dir = "tmp"

[build]
  cmd = "go build -o ./tmp/flowcode ./cmd/server"
  bin = "./tmp/flowcode server"
  full_bin = ""
  include_ext = ["go", "yaml", "yml", "toml", "json"]
  exclude_dir = ["tmp", "docs", "web", "node_modules", ".git"]
  include_dir = []
  delay = 500          # ms，防抖延迟
  stop_on_error = true # 编译失败时不重启

[log]
  time = true

[misc]
  clean_on_exit = true
```

启动方式：

```bash
# 安装 air
go install github.com/cosmtrek/air@latest

# 开发模式启动（自动热加载）
air

# 等价于：
make dev
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `delay` | 500ms | 防抖，避免连续保存触发多次重启 |
| `stop_on_error` | true | 编译错误不杀旧进程，保持服务可用 |
| `include_ext` | go/yaml/toml/json | 配置变更也触发重启 |
| `exclude_dir` | tmp/docs/web | 排除无关目录，减少误触发 |

#### 2.3.7c 数据库迁移（Gorm AutoMigrate）

使用 Gorm 的 `AutoMigrate` 根据 Go struct 自动同步数据库 schema，无需手写 SQL migration：

**策略**：

```
┌─ 服务启动 ──────────────────────────────────────────┐
│                                                       │
│  1. Gorm 连接 MySQL                                   │
│  2. AutoMigrate(&User{}, &Org{}, &Project{}, ...)    │
│  3. 自动: CREATE TABLE / ADD COLUMN / ADD INDEX      │
│  4. 打印迁移日志: "migrated table: users"            │

│                                                       │
└───────────────────────────────────────────────────────┘
```

**迁移模型注册**（启动时一次性执行所有 Model）：

```go
// internal/database/migrate.go
func AutoMigrate(db *gorm.DB) error {

        &model.OrgMember{},

        &model.GitCredential{},
        &model.Requirement{},
        &model.Issue{},
        &model.Label{},
        &model.IssueLabel{},
        &model.ExecutionLog{},
        &model.TestCase{},
        &model.TestResult{},
        &model.HistoryRecord{},
        &model.Skill{},
        &model.AIToolConfig{},
        &model.MonitorEvent{},
        &model.Device{},

```

**AutoMigrate 行为**：

| 操作 | AutoMigrate 是否支持 | 说明 |
|------|:---:|------|
| 新建表 | ✅ | 根据 struct tag 创建 |
| 新增字段 | ✅ | 自动 ADD COLUMN |
| 新增索引 | ✅ | 自动 ADD INDEX |
| 修改字段类型 | ❌ | 需手动 SQL 或 [Atlas](https://atlasgo.io/) 等工具 |
| 删除字段 | ❌ | 不会 DROP COLUMN（安全保护） |
| 重命名字段 | ❌ | 需手动迁移 |

> **生产环境建议**：AutoMigrate 适合开发/测试阶段。生产环境推荐搭配 [Atlas](https://atlasgo.io/) 或 [golang-migrate](https://github.com/golang-migrate/migrate) 生成版本化 DDL 脚本，实现可审查、可回滚的 migration。

#### 2.3.8d Webhook 签名验证

HookRegistry 对外发送 Webhook 时附带 HMAC-SHA256 签名，接收方可验证请求真实性：

```
X-FlowCode-Signature: sha256=<HMAC(payload, secret)>
X-FlowCode-Event: issue:approved
```

接收方验证（以 Node.js 为例）：

```js
const crypto = require('crypto');
const signature = req.headers['x-flowcode-signature'];
const expected = 'sha256=' + crypto.createHmac('sha256', secret).update(JSON.stringify(req.body)).digest('hex');
if (crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
  // 验证通过
}
```

| 用户密码 | bcrypt (cost=12) 哈希 |
| SSH 私钥 | AES-256-GCM 加密存储，仅在使用时解密 |
| Git PAT Token | AES-256-GCM 加密存储 |
| JWT Secret | 环境变量注入，不落地配置文件 |
| API Key | SHA-256 哈希存储（仅创建时返回明文一次） |
| 审计日志 | 仅追加写入，不可删除 |

### 2.3.9 国际化 (i18n)

支持中文 (zh-CN) 和英文 (en-US) 双语言，覆盖前端界面、后端错误信息、CLI 输出。

```
┌─ 前端 (vue-i18n) ──────────────────────────────────────┐
│  src/locales/                                           │
│  ├── zh-CN.json   ← 中文语言包 (默认)                    │
│  └── en-US.json   ← 英文语言包                           │
│                                                          │
│  切换方式: 导航栏语言选择器 → Pinia locale store         │
│            → 首次访问根据浏览器 Accept-Language 推断      │
└──────────────────────────────────────────────────────────┘

┌─ 后端 (Go i18n bundle) ─────────────────────────────────┐
│  internal/i18n/                                          │
│  ├── bundle.go    ← 消息加载器 (go-i18n)                 │
│  ├── zh-CN.toml   ← 中文错误消息                          │
│  └── en-US.toml   ← 英文错误消息                          │
│                                                          │
│  识别优先级:                                              │
│  1. 请求头 Accept-Language (中间件 i18n.go 解析)          │
│  2. 用户偏好 User.locale (登录后从 DB 读取)               │
│  3. 默认 zh-CN                                           │
└──────────────────────────────────────────────────────────┘
```

**覆盖范围**：

| 层级 | 内容 | 方式 |
|------|------|------|
| 前端 UI | 导航/按钮/表单标签/提示 | vue-i18n `$t()` 插值 |
| 后端错误码 | 所有 `{code, message}` 的 message 字段 | go-i18n `Localize()` |
| CLI 输出 | 命令帮助/状态提示/错误信息 | OS locale 检测 + 用户配置 |
| 邮件通知 | 邀请成员/异常告警/密码重置 | 模板文件 + locale 参数 |

> 用户生成内容（Issue 标题/描述、需求内容等）不做翻译，保持原语言存储。

## 2.4 SaaS 多租户架构

### 2.4.1 隔离模型：Shared DB + orgId

flowcode 采用 **共享数据库 + orgId 过滤** 的轻量多租户模型。一张 MySQL 服务所有组织，通过 Gorm Scoped 查询自动注入 `WHERE org_id = ?`。

```
┌────────────────────────────────────────────────────────┐
│                    MySQL (单实例)                        │
│                                                         │
│  ┌─────────┐ ┌──────────┐ ┌────────┐ ┌───────────┐   │
│  │ Projects │ │  Issues  │ │  Users │ │ OrgMembers│   │
│  │  org_id  │ │  org_id  │ │        │ │  org_id   │   │
│  └─────────┘ └──────────┘ └────────┘ └───────────┘   │
│                                                         │
│  Org A 的数据: WHERE org_id = 'org-aaa'                 │
│  Org B 的数据: WHERE org_id = 'org-bbb'                 │
└────────────────────────────────────────────────────────┘
```

> 面向独立开发者和小团队，Shared DB 是最优解。若未来有企业合规需求，可通过 `DB_PER_TENANT` 模式扩展。

### 2.4.2 租户识别：JWT + APIKey（无 Subdomain）

不做域名路由，直接通过请求中携带的凭证识别 orgId：

```go
// internal/middleware/tenant.go

func TenantMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        var orgID string

        // 1. JWT（Web 登录用户，claims 中携带当前 orgId）
        if claims := GetJWTClaims(c); claims != nil {
            orgID = claims.OrgID
        }

        // 2. API Key（CLI / CI 调用，key 本身绑定到 org）
        if apiKey := c.GetHeader("X-FlowCode-API-Key"); apiKey != "" {
            token, err := findApiKey(c, apiKey)
            if err == nil {
                orgID = token.OrgID
            }
        }

        // 3. 组织切换：用户属于多个组织时，通过 Header 切换
        if switchOrg := c.GetHeader("X-FlowCode-Org-Id"); switchOrg != "" {
            if isMember(c, switchOrg) {
                orgID = switchOrg
            }
        }

        if orgID == "" {
            c.AbortWithStatusJSON(401, gin.H{"error": "tenant not identified"})
            return
        }

        c.Set("orgID", orgID)
        c.Next()
    }
}
```

**识别优先级**：`X-FlowCode-Org-Id`（显式切换）> JWT claims > API Key

### 2.4.3 Gorm Scoped 查询

所有数据库查询通过统一入口自动注入 orgId 过滤，防止跨租户数据泄露：

```go
// internal/service/base.go

func (s *BaseService) DB(ctx context.Context) *gorm.DB {
    orgID := ctx.Value("orgID").(string)
    return s.db.WithContext(ctx).Where("org_id = ?", orgID)
}

// 使用：所有 Service 通过 DB(ctx) 替代 db
func (s *IssueService) List(ctx context.Context, projectID string) ([]Issue, error) {
    var issues []Issue
    // SQL: SELECT * FROM issues WHERE org_id = ? AND project_id = ?
    return issues, s.DB(ctx).Where("project_id = ?", projectID).Find(&issues).Error
}
```
