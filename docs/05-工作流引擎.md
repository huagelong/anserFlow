# 05 - 工作流引擎

## 5.1 Issue 生命周期状态机

```
                        ┌─────────┐
                        │  draft  │  ← 用户创建 / 监控自动创建 / 需求拆解
                        └────┬────┘

                    │   管理员审核      │  ← 必须审核，不可跳过
                    └────────┬────────┘

     │                 └────┬─────┘
```

## 5.2 状态详解

| 状态 | 含义 | 可触发动作 |
|------|------|-----------|
| `draft` | 草稿，待审核 | approve / reject / edit / delete |
| `approved` | 审核通过，等待生成计划并执行 | enqueue / edit / reject |
| `rejected` | 审核驳回 | reopen / edit / delete / close |
| `queued` | 已进入执行队列 | cancel |
| `running` | AI 正在执行 | cancel / pause / ask (AI主动询问) |
| `paused` | 暂停中 | continue / cancel |
| `waiting_input` | AI 暂停等待用户回答 | answer / skip_question / cancel |
| `done` | 执行成功 | create_pr / retry / close |
| `failed` | 执行失败 | retry / edit / reject / close |
| `cancelled` | 已取消 | retry / close |
| `pr_ready` | 代码就绪，待创建 PR | create_pr / retry |
| `pr_created` | PR 已创建 | (Webhook 驱动后续状态) / reopen |
| `deployed` | 已部署 | close / reopen |
| `closed` | 已关闭 | reopen（重新打开后自动回到 `approved`） |
| `reopened` | 已重新打开 | → approved（自动回到可执行入口） |

> **审核要求**：Issue 执行前**必须经过审核**（`draft` → `approve` → `approved` → `enqueue` → `running`）。`approved` 是进入计划生成与执行队列的唯一入口；执行计划可在审批后提前生成，也可在 Worker 真正开始执行前自动补齐。`close` 后 `reopen` 会先进入 `reopened`，再自动回到 `approved`，无需额外人工 `approve`。`fast_exec` 仅限 Admin 角色显式传 `force=true` 使用。
>
> **关闭/重开循环**：Issue 在 `done`、`failed`、`cancelled`、`deployed`、`rejected` 状态下均可 `close`。关闭后随时 `reopen`，重开后自动进入 `approved`，可重新生成计划并再次入队执行。

## 5.3 状态机实现 — 基于 looplab/fsm

> looplab/fsm 是 Go 生态最成熟的轻量 FSM 库，提供声明式事件定义、回调钩子、Graphviz 可视化，零运维成本。
>
> ```
> go get github.com/looplab/fsm
> ```

### 5.3.1 Issue 状态机定义

```go
// internal/fsm/issue_fsm.go

import (
    "context"
    "github.com/looplab/fsm"
    "gorm.io/gorm"
)

// IssueFSM 封装 Issue 状态机
type IssueFSM struct {
    *fsm.FSM
    db  *gorm.DB
    svc *deps  // 依赖注入容器 (见 5.3.4)
}

// NewIssueFSM 创建 Issue 状态机实例
// issue: 当前数据库中的 Issue 记录（用于获取当前状态）
func NewIssueFSM(issue *model.Issue, deps *FSMDependencies) *IssueFSM {
    f := &IssueFSM{db: deps.DB, svc: deps}

    f.FSM = fsm.NewFSM(
        issue.Status,  // 从 DB 恢复当前状态
        fsm.Events{
            //    事件名        源状态                          目标状态
            {Name: "approve",   Src: []string{"draft"},         Dst: "approved"},
            {Name: "reject",    Src: []string{"draft"},         Dst: "rejected"},
            {Name: "cancel",    Src: []string{"draft", "queued", "running", "paused"}, Dst: "cancelled"},

            // 执行链（必须 approved 后才能进入；正式执行前先确保已有计划）
            {Name: "enqueue",   Src: []string{"approved"},      Dst: "queued"},
            {Name: "pickup",    Src: []string{"queued"},        Dst: "running"},
            {Name: "pause",     Src: []string{"running"},       Dst: "paused"},
            {Name: "resume",    Src: []string{"paused"},        Dst: "running"},
            {Name: "done",      Src: []string{"running"},       Dst: "done"},
            {Name: "fail",      Src: []string{"running", "pr_ready"}, Dst: "failed"},

            // 跳过执行（opencode 判定 needsCodeChange=false，审核后直接标记完成）
            {Name: "skip",      Src: []string{"approved"},      Dst: "done"},

            // 重试 & PR
            {Name: "retry",     Src: []string{"failed", "cancelled", "done"}, Dst: "approved"},
            {Name: "mark_pr_ready", Src: []string{"done"},     Dst: "pr_ready"},
            {Name: "create_pr", Src: []string{"pr_ready"},     Dst: "pr_created"},
            {Name: "deploy",    Src: []string{"pr_created"},   Dst: "deployed"},

            // 关闭 ↔ 重新打开（可重复）
            {Name: "close",     Src: []string{"deployed", "done", "cancelled", "failed", "rejected"}, Dst: "closed"},
            {Name: "reopen",    Src: []string{"rejected", "cancelled", "closed", "done", "deployed", "pr_created"}, Dst: "reopened"},
            {Name: "reopen_to_approved", Src: []string{"reopened"}, Dst: "approved"},

            // 快捷通道：管理员跳过审核直接执行（需 Admin 角色 + 显式传参 force=true）
            {Name: "fast_exec",  Src: []string{"approved"},    Dst: "queued"},
        },
        fsm.Callbacks{
            // ---- Issue 生命周期回调 ----
            "before_approve":          f.beforeApprove,
            "after_approve":           f.afterApprove,
            "after_reject":            f.afterReject,
            "before_enqueue":          f.beforeEnqueue,
            "after_enqueue":           f.afterEnqueue,
            "before_pickup":           f.beforePickup,
            "after_pickup":            f.afterPickup,
            "after_done":              f.afterDone,
            "after_fail":              f.afterFail,
            "after_retry":             f.afterRetry,
            "after_cancel":            f.afterCancel,
            "before_create_pr":        f.beforeCreatePR,
            "after_create_pr":         f.afterCreatePR,
            "after_deploy":            f.afterDeploy,
            "after_close":             f.afterClose,
            "after_reopen":            f.afterReopen,
            "after_reopen_to_approved": f.afterReopenToApproved,
            "after_skip":              f.afterSkip,      // 跳过执行回调（发射 event + 审计）

            // ---- 通用回调 ----
            "after_event":             f.afterAnyEvent,     // 所有事件完成后触发
            "leave_state":             f.leaveState,        // 离开任意状态
            "enter_state":             f.enterState,        // 进入任意状态
        },
    )
    return f
}
```

### 5.3.3 状态持久化与恢复

```go
// internal/fsm/persistence.go

// SaveState 将 fsm 当前状态写回 Issue 记录
func (f *IssueFSM) SaveState(ctx context.Context, issue *model.Issue) error {
    issue.Status = f.Current()
    return f.db.WithContext(ctx).Model(issue).Update("status", issue.Status).Error
}

// RestoreFSM 从数据库恢复 FSM 实例
func RestoreIssueFSM(ctx context.Context, db *gorm.DB, deps *FSMDependencies, issueID string) (*IssueFSM, *model.Issue, error) {
    var issue model.Issue
    if err := db.WithContext(ctx).First(&issue, "id = ?", issueID).Error; err != nil {
        return nil, nil, err
    }
    return NewIssueFSM(&issue, deps), &issue, nil
}

// Transition 执行状态转换 + 自动持久化
// 对外暴露的统一入口，业务层只需调用此方法
func (f *IssueFSM) Transition(ctx context.Context, issue *model.Issue, event string) error {
    if err := f.Event(ctx, event); err != nil {
        return fmt.Errorf("fsm: %w", err)
    }
    return f.SaveState(ctx, issue)
}
```

## 5.4 关键业务逻辑 — 回调实现

原有的审核、调度、重试等逻辑全部迁移到 fsm 回调中，保持业务逻辑与状态转换解耦。

### 5.4.1 审核流程 (draft → approved / rejected)

```go
// internal/fsm/issue_callbacks.go

// beforeApprove — 审核前置守卫
func (f *IssueFSM) beforeApprove(ctx context.Context, e *fsm.Event) {
    // 操作者必须是 admin
    userID, _ := ctx.Value("userID").(string)
    userRole, _ := ctx.Value("userRole").(string)
    if userRole != "admin" {
        e.Cancel(fmt.Errorf("only admin can approve"))
        return
    }

    // 项目必须已配置 Git 凭证
    project, _ := f.svc.DB.GetProject(ctx, e.Args[0].(string))  // projectID
    if project.GitRepoUrl == "" {
        e.Cancel(fmt.Errorf("project git URL not configured"))
        return
    }
    e.Args = append(e.Args, project) // 传递给 after_approve
}

func (f *IssueFSM) afterApprove(ctx context.Context, e *fsm.Event) {
    project := e.Args[0].(*model.Project)
    issue := e.Args[1].(*model.Issue)

    // 自动执行：project.settings.autoExecute === true
    if project.Settings.AutoExecute {
        f.Event(ctx, "enqueue")  // 衔接执行
    }

    // 发射 Hook
    f.svc.HookRegistry.Emit(ctx, "issue:approved", issue)
}

// afterReject — 审核驳回
func (f *IssueFSM) afterReject(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)
    reason := e.Args[1].(string)  // 驳回原因

    // 记录驳回原因
    f.db.Model(issue).Update("reject_reason", reason)

    // 发射 Hook
    f.svc.HookRegistry.Emit(ctx, "issue:rejected", issue, reason)
}
```

### 5.4.2 执行调度 (approved → queued → running)

```go
// beforeEnqueue — 入队前置检查
func (f *IssueFSM) beforeEnqueue(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)

    // 校验前置分析
    if !issue.AnalysisCompleted {
        e.Cancel(fmt.Errorf("issue analysis not completed"))
        return
    }

    // 校验 AI 工具配置
    tool, err := f.svc.DB.GetAIToolConfig(ctx, issue.ProjectID, issue.AITool)
    if err != nil || !tool.Enabled {
        e.Cancel(fmt.Errorf("AI tool not configured: %s", issue.AITool))
        return
    }
    e.Args = append(e.Args, tool)
}

// afterEnqueue — 投递异步任务
func (f *IssueFSM) afterEnqueue(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)

    // 创建执行日志
    log := &model.ExecutionLog{
        IssueID: issue.ID,
        Status:  "queued",
        Attempt: issue.Attempt + 1,
    }
    f.db.Create(log)

    // 投递 Asynq 任务
    task, _ := asynq.NewTask("issue:execute", marshal(ExecutePayload{
        IssueID: issue.ID,
        LogID:   log.ID,
    }), asynq.MaxRetry(0))
    f.svc.AsynqClient.Enqueue(task)

    f.svc.HookRegistry.Emit(ctx, "issue:queued", issue)
}

// beforePickup — Worker 拾取任务时触发
func (f *IssueFSM) beforePickup(ctx context.Context, e *fsm.Event) {
    // 可选：检查并发数限制
    f.svc.HookRegistry.Emit(ctx, "issue:execution:start", e.Args[0])
}
```

Worker 侧的状态驱动 (Asynq handler)：

```go
// internal/queue/handler.go

func ExecuteIssueHandler(ctx context.Context, t *asynq.Task) error {
    var payload ExecutePayload
    json.Unmarshal(t.Payload(), &payload)

    // 1. 恢复状态机
    fsm, issue, _ := RestoreIssueFSM(ctx, db, deps, payload.IssueID)

    // 2. 前置检查: opencode 分析结果
    if issue.AnalysisCompleted && !issue.NeedsCodeChange {
        // 无需代码变更（纯文档/配置/沟通类 Issue），直接标记完成
        slog.Info("Issue needs no code change, auto-completing",
            "issue", issue.ID, "analysis", issue.AnalysisReason)
        fsm.Transition(ctx, issue, "skip")
        fsm.SaveState(ctx, issue)
        return nil
    }

    // 3. 生成执行计划（若尚未生成）
    if !issue.PlanCompleted {
        plan, err := deps.AIToolService.Plan(ctx, issue)
        if err != nil {
            return err
        }
        issue.PlanSummary = plan.Summary
        issue.PlanDetail = plan.Detail
        issue.PlanCompleted = true
        issue.PlanGeneratedAt = time.Now()
        db.Save(issue)
        deps.HookRegistry.Emit(ctx, HookIssuePlanned, issue)
    }

    // 4. 入队 → 执行中
    fsm.Transition(ctx, issue, "pickup")

    // 5. 调用 AI 工具
    result, err := deps.AIToolService.Execute(ctx, issue)

    // 5. 根据结果驱动状态
    if err != nil {
        fsm.Event(ctx, "fail")
    } else {
        fsm.Event(ctx, "done")
    }
    fsm.SaveState(ctx, issue)
    return nil
}
```

### 5.4.3 继续编辑 (vibe coding 续跑)

```go
// POST /api/v1/issues/:id/continue
func (h *IssueHandler) Continue(c *gin.Context) {
    fsm, issue, _ := RestoreIssueFSM(c.Request.Context(), h.db, h.deps, c.Param("id"))

    // done / failed / paused → 重新进入 approved 再入队
    if fsm.InState("done", "failed", "paused") {
        fsm.Transition(ctx, issue, "retry")
        fsm.Transition(ctx, issue, "enqueue")
    }
    // 保留原 gitBranch，AI 在原分支基础上继续
    c.JSON(200, issue)
}
```

### 5.4.4 PR 创建 (done → pr_ready → pr_created)

```go
func (f *IssueFSM) beforeCreatePR(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)

    // 前置: 必须有 Git 凭证
    cred, err := f.svc.DB.GetGitCredential(ctx, issue.ProjectID)
    if err != nil {
        e.Cancel(fmt.Errorf("git credential not found"))
        return
    }

    // 提交变更
    commitSha, _ := f.svc.GitService.CommitChanges(ctx, issue.GitBranch, issue)
    issue.GitCommitSha = commitSha
    e.Args = append(e.Args, cred)
}

func (f *IssueFSM) afterCreatePR(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)

    // 创建 PR
    pr, _ := f.svc.GitService.CreatePR(ctx, issue)
    issue.GitPRUrl = pr.URL
    f.db.Save(issue)

    f.svc.HookRegistry.Emit(ctx, "issue:pr:created", issue, pr)
}

// 简化端到端流程
func (f *IssueFSM) AutoCreatePRPipeline(ctx context.Context, issue *model.Issue) error {
    if err := f.Transition(ctx, issue, "mark_pr_ready"); err != nil {
        return err
    }
    return f.Transition(ctx, issue, "create_pr")
}
```

### 5.4.5 失败重试与回退

```go
// internal/fsm/issue_callbacks.go

func (f *IssueFSM) afterFail(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)
    project, _ := f.svc.DB.GetProject(ctx, issue.ProjectID)

    maxRetries := project.Settings.MaxRetries  // 默认 3
    if issue.Attempt < maxRetries {
        // 自动降级重试
        f.Event(ctx, "retry")
        f.Event(ctx, "enqueue")
        f.svc.HookRegistry.Emit(ctx, "issue:execution:retry", issue, issue.Attempt)
        return
    }

    // 超过最大重试 → 保持在 failed，通知管理员
    f.svc.HookRegistry.Emit(ctx, "issue:execution:failed", issue)
}
```

## 5.5 工作流 Hook 点 (集成到 FSM 回调)

Hook 事件不再单独维护列表，而是直接在 fsm 回调中 emit。事件常量保留用于外部订阅：

```go
// internal/fsm/hook_events.go

const (
    // Issue 事件
    HookIssueCreated        = "issue:created"
    HookIssueAnalyzed       = "issue:analyzed"       // opencode 前置分析完成
    HookIssueClassified     = "issue:classified"      // AI 分类完成回写
    HookIssueApproved       = "issue:approved"
    HookIssuePlanned        = "issue:planned"        // 执行计划生成完成
    HookIssueRejected       = "issue:rejected"
    HookIssueQueued         = "issue:queued"
    HookIssueExecutionStart = "issue:execution:start"
    HookIssueExecutionLog   = "issue:execution:log"
    HookIssueExecutionDone  = "issue:execution:done"
    HookIssueExecutionFail  = "issue:execution:failed"
    HookIssueExecutionRetry = "issue:execution:retry"
    HookIssuePRCreated      = "issue:pr:created"
    HookIssuePRMerged       = "issue:pr:merged"
    HookIssueDeployed       = "issue:deployed"
    HookIssueClosed         = "issue:closed"
    HookIssueSkipped        = "issue:skipped"       // 无需代码变更，跳过执行
```

TestCase 必须先经过审核，才能调用 AI 工具生成测试代码：

```
                       │ pending_review│

              ▼               │               ▼

        └────┬─────┘         │         └──────────┘
             │               │
             ▼               │
       ┌────────────┐        │

       └─────┬──────┘        │
             │               │
             ▼               │
```

### 5.7.1 TestCase FSM 实现

> **设计说明**：TestCase 使用两个独立字段管理不同阶段：
> - `reviewStatus`：审核视角，记录当前审核结论 (draft → pending_review → approved / rejected)
> - `status`：完整生命周期状态，覆盖审核与执行阶段 (draft → pending_review → approved → generating → generated → passed / failed / rejected)
> 
> FSM 使用 `status` 字段初始化，因为它覆盖了完整生命周期。审核状态变更通过回调同步更新 `reviewStatus` 字段。

```go
// internal/fsm/test_case_fsm.go

type TestCaseFSM struct {
    *fsm.FSM
    db  *gorm.DB
    svc *FSMDependencies
}

func NewTestCaseFSM(tc *model.TestCase, deps *FSMDependencies) *TestCaseFSM {
    f := &TestCaseFSM{db: deps.DB, svc: deps}

    f.FSM = fsm.NewFSM(
        tc.Status,  // 使用 status 字段初始化，覆盖完整生命周期
        fsm.Events{
            // 审核流程: 同时更新 reviewStatus 和 status
            {Name: "submit",    Src: []string{"draft"},             Dst: "pending_review"},
            {Name: "approve",   Src: []string{"pending_review"},    Dst: "approved"},
            {Name: "reject",    Src: []string{"pending_review"},    Dst: "rejected"},
            
            // 执行流程: 只更新 status
            {Name: "generate",  Src: []string{"approved"},          Dst: "generating"},
            {Name: "generated", Src: []string{"generating"},        Dst: "generated"},
            {Name: "pass",      Src: []string{"generated"},         Dst: "passed"},
            {Name: "fail",      Src: []string{"generated"},         Dst: "failed"},
            
            // 重试: 从 failed/rejected 回到 approved
            {Name: "retry",     Src: []string{"failed", "rejected"}, Dst: "approved"},
        },
        fsm.Callbacks{
            "before_approve": f.beforeTestCaseApprove,
            "after_approve":  f.afterTestCaseApprove,   // 自动触发 AI 生成
            "after_reject":   f.afterTestCaseReject,
            "after_generate": f.afterGenerate,
            "after_generated": f.afterGenerated,
            "after_pass":     f.afterPassOrFail,
            "after_fail":     f.afterPassOrFail,
            "after_event":    f.afterTestCaseEvent,      // 通用: 记录历史 + emit hook
        },
    )
    return f
}
```

### 5.7.2 审核回调 — 自动衔接 AI 生成

```go
// internal/fsm/test_case_callbacks.go

func (f *TestCaseFSM) beforeTestCaseApprove(ctx context.Context, e *fsm.Event) {
    // admin 权限守卫
    role, _ := ctx.Value("userRole").(string)
    if role != "admin" {
        e.Cancel(fmt.Errorf("only admin can approve"))
    }
}

// afterTestCaseApprove — 审核通过后自动触发 AI 生成
func (f *TestCaseFSM) afterTestCaseApprove(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)

    // 同步更新 reviewStatus 字段
    f.db.Model(tc).Update("reviewStatus", "approved")

    f.svc.HookRegistry.Emit(ctx, HookTestCaseApproved, tc)

    // 自动衔接: approved → generating
    go func() {
        _ = f.Event(context.Background(), "generate")
        // generate 触发 Asynq 任务调用 AI
    }()
}

// afterTestCaseReject — 审核驳回
func (f *TestCaseFSM) afterTestCaseReject(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)

    // 同步更新 reviewStatus 字段
    f.db.Model(tc).Update("reviewStatus", "rejected")

    f.svc.HookRegistry.Emit(ctx, HookTestCaseRejected, tc)
}

// afterGenerate — 投递 AI 生成任务
func (f *TestCaseFSM) afterGenerate(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)

    task, _ := asynq.NewTask("test_case:generate", marshal(GenTestPayload{
        TestCaseID:  tc.ID,
        Description: tc.Description,
        IssueID:     tc.IssueID,
    }), asynq.MaxRetry(0))
    f.svc.AsynqClient.Enqueue(task)
}

// afterGenerated — AI 生成完成后自动运行测试
func (f *TestCaseFSM) afterGenerated(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)

    f.svc.HookRegistry.Emit(ctx, HookTestCaseGenerated, tc)

    // 自动运行测试（Docker 沙盒内执行）
    result, err := f.svc.TestRunner.Run(ctx, tc)
    if err != nil {
        // runner 异常，标记失败但记录错误
        f.db.Model(tc).Update("result", map[string]any{
            "status": "failed",
            "errors": []map[string]any{{"message": err.Error()}},
            "runAt":  time.Now(),
        })
        f.Event(ctx, "fail")
        return
    }

    // 写入结果
    f.db.Model(tc).Updates(map[string]any{
        "status": result.Status,
        "result": result,
    })

    if result.Status == "passed" {
        f.Event(ctx, "pass")
    } else {
        f.Event(ctx, "fail")
    }
}

func (f *TestCaseFSM) afterPassOrFail(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)
    hookName := HookTestCasePassed
    if e.Event == "fail" {
        hookName = HookTestCaseFailed
    }
    f.svc.HookRegistry.Emit(ctx, hookName, tc)
}
```

### 5.7.3 完整审核→生成→测试流程 API 层

## 5.8 历史记录自动采集

历史记录通过 **fsm 的 `after_event` / `enter_state` 回调** 统一采集，替代原来分散在多个 Gorm Hook 中的逻辑。

### 5.8.1 通用历史记录回调

```go
// internal/fsm/history_hook.go

// afterAnyEvent — 所有 fsm 事件结束后自动记录历史
func (f *IssueFSM) afterAnyEvent(ctx context.Context, e *fsm.Event) {
    // 记录状态变更
    _ = f.db.WithContext(ctx).Create(&model.HistoryRecord{
        TargetType: "issue",
        TargetId:   e.Args[0].(*model.Issue).ID,
        Action:     "status_changed",
        FieldName:  "status",
        OldValue:   mustJSON(map[string]string{"status": e.Src}),
        NewValue:   mustJSON(map[string]string{"status": e.Dst}),
        ChangedBy:  ctx.Value("userID").(string),
    }).Error
}

// enterState — 进入特定状态时的附加记录
func (f *IssueFSM) enterState(ctx context.Context, e *fsm.Event) {
    issue := e.Args[0].(*model.Issue)
    // 例如进入 deployed 状态额外记录
    if e.Dst == "deployed" {
        f.svc.HookRegistry.Emit(ctx, HookIssueDeployed, issue)
    }
}

// leaveState — 离开状态时的清理逻辑
func (f *IssueFSM) leaveState(ctx context.Context, e *fsm.Event) {
    // 离开 running 状态时，关闭 WebSocket 连接等
    if e.Src == "running" {
        f.svc.WsHub.CloseConn(e.Args[0].(*model.Issue).ID)
    }
}

// TestCase 使用 after_event 统一记录状态变更
// 注意: 状态变更记录到 status 字段，审核状态变更通过回调同步更新 reviewStatus
func (f *TestCaseFSM) afterTestCaseEvent(ctx context.Context, e *fsm.Event) {
    tc := e.Args[0].(*model.TestCase)
    
    // 记录主状态变更
    _ = f.db.WithContext(ctx).Create(&model.HistoryRecord{
        TargetType: "test_case",
        TargetId:   tc.ID,
        Action:     "status_changed",
        FieldName:  "status",
        OldValue:   mustJSON(map[string]string{"status": e.Src}),
        NewValue:   mustJSON(map[string]string{"status": e.Dst}),
        ChangedBy:  ctx.Value("userID").(string),
    }).Error
    
    // submit 事件需要同步更新 reviewStatus (其他审核事件在专门的回调中处理)
    if e.Event == "submit" {
        f.db.Model(tc).Update("reviewStatus", "pending_review")
        
        _ = f.db.WithContext(ctx).Create(&model.HistoryRecord{
            TargetType: "test_case",
            TargetId:   tc.ID,
            Action:     "status_changed",
            FieldName:  "reviewStatus",
            OldValue:   mustJSON(map[string]string{"reviewStatus": "draft"}),
            NewValue:   mustJSON(map[string]string{"reviewStatus": "pending_review"}),
            ChangedBy:  ctx.Value("userID").(string),
        }).Error
    }
}
```

### 5.8.2 字段级变更仍然使用 Gorm Hook

状态变更由 fsm `after_event` 处理；标题/描述等字段的修改仍然用 Gorm Hook（见 [03-数据模型.md 5.8 原实现]）。两者互不冲突：

```
[Issue 状态变更]  → fsm after_event    → HistoryRecord (status_changed)
[Issue 字段修改]  → Gorm AfterUpdate   → HistoryRecord (updated, per field)
[Issue 创建]      → Gorm AfterCreate   → HistoryRecord (created)
```

### 5.8.3 注册

Gorm AutoMigrate 时自动绑定 Gorm Hook；fsm 回调在 `NewIssueFSM()` / `NewTestCaseFSM()` 初始化时注册。

### Webhook 防重入

```go
func (f *IssueFSM) Transition(ctx context.Context, issue *model.Issue, event string) error {
    // 乐观锁: 仅当 DB 中的状态与 fsm 当前状态一致时才执行
    result := f.db.WithContext(ctx).
        Model(issue).
        Where("status = ?", f.Current()).
        Update("status", fsm.StateTo(e))  // 预判目标状态

    if result.RowsAffected == 0 {
        return fmt.Errorf("concurrent state change detected, retry")
    }

    // 更新成功，本地 fsm 同步
    return f.Event(ctx, event)
}
```

## 5.10 需求拆解流程 (AI Decompose)

需求（Requirement）不使用 FSM，通过状态字段 `status` 控制生命周期。从 `approved` 到 `in_progress` 之间增加 AI 拆解环节：

```
需求创建 (draft)
    │
    ▼
  审核 (reviewing → approved)
    │
    ▼
  POST /api/v1/requirements/:id/ai-decompose
    │
    ▼
  投递 Asynq → opencode decompose
    │  (分析需求标题+描述+技术栈 → 输出子 Issue 列表)
    ▼
  前端展示拆解结果
    │
    ├── 用户勾选/调整 → POST .../convert-to-issues
    │       │
    │       ▼
    │  批量创建 Issue (draft) → status = in_progress
    │
    └── 用户取消/重新拆解 → 可再次调用 ai-decompose
```

**拆解阶段状态映射**：

| Requirement.status | 含义 |
|--------------------|------|
| `draft` | 草稿，可编辑 |
| `reviewing` | 审核中 |
| `approved` | 已审核通过，**等待 AI 拆解** |
| `in_progress` | 已拆解为 Issue，Issue 执行中 |
| `done` | 全部 Issue 完成 |
| `archived` | 归档 |

> `decomposeResult` 字段缓存最近一次 AI 拆解结果，用户重新打开时可恢复上次的勾选状态。
