# 09 - 监控与反馈闭环

## 9.1 设计理念

flowcode 的监控系统不是又一个 APM 工具，而是一个 **轻量级的问题回流管道**。目标是让线上发生的每一个异常都有机会自动变成 Issue，进入 vibe coding 的修复循环。

```
线上错误 → 自动收集 → 智能去重 → 分类分级 → 自动创建 Issue → AI 修复 → PR → 部署
    ↑                                                                              │
    └──────────────────────── 反馈闭环 ────────────────────────────────────────────┘
```

## 9.2 监控事件接入方式

### 方式一：内置前端 SDK（推荐）

```html
<!-- 部署到线上的项目只需引入一行脚本 -->
<script src="https://flowcode.example.com/sdk/flowcode-monitor.min.js"></script>
<script>
  FlowcodeMonitor.init({
    projectKey: 'fc_proj_abc123',
    endpoint: 'https://flowcode.example.com',
    captureUnhandledErrors: true,
    captureConsoleErrors: true,
    captureFetchErrors: true,
    sampleRate: 1.0,                    // 采样率
    ignorePatterns: [                   // 忽略规则
      /ResizeObserver loop/,
      /Script error\.?$/,
    ],
    beforeSend: (event) => {
      // 可以在发送前过滤或修改事件
      if (event.message.includes('test')) return null;
      return event;
    },
  });
</script>
```

### 方式二：服务端 SDK

```go
// Go 项目中使用
import "github.com/flowcode/go-sdk/monitor"

monitor := monitor.New(monitor.Config{
    ProjectKey:  "fc_proj_abc123",
    Endpoint:    "https://flowcode.example.com",
    Environment: "production",
    Release:     "v1.3.2",
})

// 手动上报
app.Use(func(c *gin.Context) {
    defer func() {
        if err := recover(); err != nil {
            monitor.CaptureError(fmt.Errorf("%v", err), monitor.CaptureContext{
                URL:    c.Request.URL.String(),
                Method: c.Request.Method,
            })
        }
    }()
    c.Next()
})
```

## 9.3 事件处理管道

```
[接收事件] → [事件验证] → [去重检查] → [优先级推断] → [分类] → [聚合] → [创建Issue?]
                  │            │            │           │         │
                  ▼            ▼            ▼           ▼         ▼
              格式校验      Hash比对    规则匹配    规则引擎   阈值判断
              限流保护     时间窗口    频率权重    关键词匹配  手动/自动
```

### 去重算法

```go
// internal/service/monitor/dedup.go

var (
    uuidRe = regexp.MustCompile(`[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}`)
    urlRe  = regexp.MustCompile(`/[a-z0-9_-]+`)
    numRe  = regexp.MustCompile(`\d+`)
)

func computeDedupHash(projectID, message, stackTrace string) string {
    // 基于 message + stackTrace 前3帧 + 项目ID 生成 SHA256
    normalized := strings.ToLower(message)
    normalized = uuidRe.ReplaceAllString(normalized, "<UUID>")
    normalized = urlRe.ReplaceAllString(normalized, "/<param>") // URL参数化
    normalized = numRe.ReplaceAllString(normalized, "<NUM>")    // 数字参数化

    topFrames := getTopFrames(stackTrace, 3)
    key := fmt.Sprintf("%s:%s:%s", projectID, normalized, topFrames)

    h := sha256.Sum256([]byte(key))
    return hex.EncodeToString(h[:])
}

func getTopFrames(stackTrace string, n int) string {
    lines := strings.Split(strings.TrimSpace(stackTrace), "\n")
    if len(lines) > n {
        lines = lines[:n]
    }
    return strings.Join(lines, "\n")
}
```

### 优先级推断规则

| 条件 | 优先级 |
|------|--------|
| 1 小时内出现 > 50 次相同事件 | P0 |
| 事件类型为 `error` 且影响核心 API | P1 |
| 事件类型为 `error` 且为边缘路径 | P2 |
| 事件类型为 `warning` | P3 |
| 用户手动标记为高优 | 按标记 |

```go
// internal/service/monitor/priority.go

func inferPriority(event MonitorEvent, recentCount int, settings ProjectSettings) Priority {
    // 1小时内高频 = P0
    if recentCount >= settings.MonitorHighThreshold {
        return PriorityP0
    }

    // 核心API错误 = P1
    if event.EventType == "error" && isCoreAPI(event.Metadata.URL) {
        return PriorityP1
    }

    // 一般错误 = P2
    if event.EventType == "error" {
        return PriorityP2
    }

    // warning = P3
    return PriorityP3
}
```

## 9.4 自动创建 Issue 策略

### 触发条件（可配置）

```json
{
  "monitor": {
    "autoCreateIssue": true,
    "minOccurrences": 3,            // 最少出现次数
    "occurrenceWindow": 3600,       // 统计窗口(秒)
    "cooldownPerEvent": 86400,      // 同一事件冷却期(秒)
    "requireStackTrace": true,      // 必须有堆栈
    "excludePatterns": [            // 排除模式
      "Network request failed",
      "Script error"
    ]
  }
}
```

### 自动生成 Issue 内容

```go
// internal/service/monitor/issue_builder.go

func buildIssueFromEvent(event MonitorEvent, recentCount int, settings ProjectSettings) *CreateIssueInput {
    metadata := event.Metadata
    return &CreateIssueInput{
        Title: fmt.Sprintf("[Monitor] %s: %s", event.EventType, truncate(event.Message, 100)),
        Description: fmt.Sprintf(`
## 监控自动创建

### 错误信息
%s

### 堆栈跟踪
`+"```"+`
%s
`+"```"+`

### 上下文
| 字段 | 值 |
|------|-----|
| 来源 | %s |
| URL | %s |
| 方法 | %s |
| 环境 | %s |
| 版本 | %s |

### 统计
- 最近 1 小时出现次数: %d
- 首次出现: %s
- 最近出现: %s

---
> 此 Issue 由 flowcode 监控系统自动创建
> 事件ID: %s
`, event.Message,
            orDefault(event.StackTrace, "无堆栈信息"),
            event.Source,
            orDefault(metadata["url"], "-"),
            orDefault(metadata["method"], "-"),
            orDefault(metadata["environment"], "-"),
            orDefault(metadata["releaseVersion"], "-"),
            recentCount,
            event.FirstSeenAt.Format(time.RFC3339),
            event.LastSeenAt.Format(time.RFC3339),
            event.ID,
        ),
        Priority: inferPriority(event, recentCount, settings),
        Category: "bug",
        Tags:     []string{"monitor", fmt.Sprintf("source:%s", event.Source)},
        Source:   "monitor",
    }
}

func truncate(s string, maxLen int) string {
    runes := []rune(s)
    if len(runes) <= maxLen {
        return s
    }
    return string(runes[:maxLen]) + "..."
}

func orDefault(val, defaultVal string) string {
    if val == "" {
        return defaultVal
    }
    return val
}
```

## 9.6 监控 SDK 事件类型

| eventType | 含义 | 自动创建 Issue |
|-----------|------|:---:|
| `error` | JavaScript 未捕获异常 | 是 |
| `unhandledrejection` | Promise 未处理拒绝 | 是 |
| `console.error` | console.error 调用 | 是 |
| `fetch-error` | fetch/XHR 请求失败 (4xx/5xx) | 是 |
| `warning` | 警告信息 | 阈值触发 |
| `perf` | 性能指标异常 (FCP/LCP 超标) | 否 |
| `feedback` | 用户主动反馈 | 手动 |

## 9.7 反馈入口

在已部署的 Web 应用中嵌入反馈按钮，用户可直接提交反馈：

```html

     data-position="bottom-right"
     data-theme="light">

```

用户提交的反馈通过 Monitor API 上报，在 flowcode 面板中显示，支持一键转为 Issue。

## 9.8 监控与 Issue 双向追溯

```
MonitorEvent ──→ Issue (自动创建)
     │                │
     │                ├── ExecutionLog
     │                ├── Git PR
     │                └── Deploy
     │
     └── 修复上线后，同名 dedupHash 事件自动标记为 "已修复"
```

修复部署后，系统检测到某 `dedupHash` 的事件不再出现，自动将相关 Issue 标记为可关闭，形成完整闭环。
