# Docs Wiki Builder

## 目标

把项目根目录的 `docs/*.md` 组织成 `wiki/` 下的细粒度知识库：

- 只以 `docs` 目录中的 Markdown 为内容来源。
- 不修改 `docs` 原文，不改动代码、配置或其他目录。
- 尽量拆成适合检索的小文档，同时保留原文语义连续性。
- 用 `wiki/README.md` 作为总目录，把小文档、主题路径和关系图谱串起来。
- 在每个小文档中加入必要的来源、父子、前后、相关主题链接，方便大模型顺链检索。

如果本 Skill 位于 `wiki/docs-wiki-builder/`，生成 wiki 内容时不要修改或覆盖该 Skill 目录。

## 执行流程

1. 读取 `docs/*.md`，按文件名顺序处理。
2. 解析 Markdown 标题时忽略代码块内的 `#`、`##`、`###`。
3. 以每个源文档的一级标题作为主题入口，在 `wiki/<源文档名>/README.md` 生成章节索引。
4. 拆分规则：
   - 没有三级标题的二级章节：拆成一个小文档。
   - 有三级标题的二级章节：二级章节生成一个 `README.md`，三级章节各自生成小文档。
   - 不要把表格、代码块、连续流程图说明切断。
   - 原章节正文尽量原样迁移，只允许添加导航元信息。
5. 生成 `wiki/README.md`：
   - 快速入口：链接到所有源文档主题入口。
   - 推荐检索路径：按业务链路组织，如产品概述 -> 系统架构 -> 数据模型 -> API -> 工作流。
   - 关系图谱：用 Mermaid 或关系表表达文档之间的依赖。
   - 细分文档树：列出所有小文档链接。
6. 为每个小文档添加导航块：
   - 来源：原始 `docs` 文件和章节位置。
   - 位置：总目录 -> 源文档 -> 父章节 -> 当前章节。
   - 相邻：上一篇、下一篇。
   - 相关主题：来自关系矩阵或标题/实体匹配的交叉链接。

## 推荐关系矩阵

优先使用 `docs` 原有编号和主题建立关系，不凭空扩展内容：

- `01-项目概述` -> `02-系统架构`、`03-数据模型`、`04-API设计`

## 小文档格式

每个小文档使用这个结构，导航块可以根据实际链接裁剪：

```md
> 来源：`docs/xx-xxx.md` 第 N 行
> 位置：[总目录](../README.md) -> [源文档](README.md) -> 当前章节
> 相邻：[上一篇](...) · [下一篇](...)
> 相关主题：[...](...) · [...](...)

## 原章节标题

原章节正文保持不变。
```

如果用户明确要求“完全原文、不要任何新增说明”，则只保留原章节正文，把导航关系集中放在 `wiki/README.md`。

## 校验

完成后检查：

- `git diff -- docs` 为空。
- `git diff --stat` 只出现 `wiki/`。
- `wiki/README.md` 能链接到所有主题入口和小文档。
- 小文档中的相对链接都能指向已生成文件。
- 代码块没有因为标题解析被错误拆开。

## 禁止事项

- 不要改写、润色或扩写 `docs` 的正文内容。
- 不要生成 `wiki` 之外的临时文件、总结文档或脚本，除非用户明确要求。
- 不要为了“更完整”补充 `docs` 没有写到的设计结论。
- 不要重排原文段落顺序；只做章节级拆分和链接组织。
