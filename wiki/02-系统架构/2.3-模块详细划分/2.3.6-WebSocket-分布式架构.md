Asynq Worker 在 Server B 执行完成 → 调用 `Hub.Publish(issueId, logLine)` 3. Server B Redis `PUBLISH exec:iss_abc "log line"` 4. 所有 Server 的 Hub 收到 → 转发给本地订阅该 `issueId` 的 WS 客户端  #### 代码结构  ```go // ws/hub.go — 每实例单例  type Hub struct {     // 房间: issueId → 客户端集合     rooms      map[string]map[*Client]bool     register   chan *Client     unregister chan *Client      // Redis Pub/Sub     rdb       *redis.Client     pubsub    *redis.PubSub     // 本地消息广播     broadcast chan Message }  func (h *Hub) Run() {     // 订阅 Redis pattern: exec:*     h.pubsub = h.rdb.PSubscribe(ctx, "exec:*")     go h.listenRedis()      for {         select {         case client := <-h.register:             h.addToRoom(client)         case client := <-h.unregister:             h.removeFromRoom(client)         case msg := <-h.broadcast:             h.sendToRoom(msg.IssueID, msg.Data)         }     } }  func (h *Hub) listenRedis() {     for msg := range h.pubsub.Channel() {         // exec:iss_abc → issueId = "iss_abc"         issueId := strings.TrimPrefix(msg.Channel, "exec:")         h.broadcast <- Message{IssueID: issueId, Data: []byte(msg.Payload)}     } }  // 发布消息到 Redis（任何实例调用） func (h *Hub) Publish(issueId string, data []byte) {     h.rdb.Publish(ctx, "exec:"+issueId, data) } ```  ```go // ws/client.go — 每连接一个 goroutine 对  type Client struct {     conn    *websocket.Conn     issueId string     send    chan []byte   // 写缓冲     hub     *Hub }  func (c *Client) readPump() {     // 读客户端消息（心跳 pong），30s 无消息则断开 }  func (c *Client) writePump() {     // 从 send channel 写入 WebSocket，30s 发一次 ping } ```  ```go // ws/message.go  type Message struct {     IssueID string `json:"issueId"`     Type    string `json:"type"`    // stdout / stderr / status / error     Data    []byte `json:"data"` } ```  #### Nginx 分布式配置  多实例部署时，Nginx 通过 `ip_hash` 保证同一客户端始终连同一实例（维持 WebSocket 长连接），但消息通过 Redis 跨实例投递：  ```nginx upstream flowcode_server {     ip_hash;                          # 会话保持     server flowcode-1:3000;     server flowcode-2:3000;     server flowcode-3:3000; }  server {     location /ws/ {         proxy_pass http://flowcode_server;         proxy_http_version 1.1;         proxy_set_header Upgrade $http_upgrade;         proxy_set_header Connection "upgrade";         proxy_read_timeout 3600s;     # 长连接超时         proxy_send_timeout 3600s;     } } ```  > `ip_hash` 仅保证连接粘性；跨实例消息投递由 Redis Pub/Sub 保证。即使 `ip_hash` 失效导致客户端重连到另一实例，消息也不会丢失。  #### 技术对比  | 方案 | 优点 | 缺点 | 适用 | |------|------|------|------| | **Redis Pub/Sub** ✅ | 零额外组件、复用现有 Redis | 消息不持久化 | flowcode（日志流，丢失可接受） | | Centrifugo | 功能全、自带鉴权/通道 | 额外部署维护 | 消息不可丢失场景 | | NATS | 高性能、持久化 | 额外组件 | 高吞吐场景 | | Kafka | 持久化、重放 | 过重 | 审计日志场景 |  **选型理由**：执行日志是实时流，断线后重连通过 `GET /api/v1/executions/:logId/stdout` 拉历史补全即可，无需消息持久化。Redis Pub/Sub 零成本满足需求。
