# Docs Wiki Todo Planner

## 使用目标

把项目已有 Markdown 文档整理成可检索的 wiki 知识库，并基于 wiki 总目录生成可执行的 vibe coding 任务计划。

支持三种模式：

- **生成 wiki**：从源 Markdown 文档目录生成或更新 wiki。
- **生成 todo list**：从 wiki 索引生成任务计划文档。
- **综合流程**：先生成 wiki，再基于刚生成的 wiki 生成任务计划。

必须遵守：

- 输出使用中文，除非用户明确要求其他语言。
- 不修改源文档原文。
- 不改动代码、配置和无关目录。
- 不创建临时文件或额外总结文档。
- 只创建或更新用户要求的目标文件/目录。
- 所有生成的链接必须指向真实存在的目标文件；写入后要校验。

## 路径策略

不要把目录写死。把路径视为可配置参数：

```text
sourceDocsDir   源 Markdown 文档目录
wikiDir         wiki 输出目录
wikiIndex       wiki 总目录文件
todoOutput      任务计划输出文件
```

推断顺序：

1. 优先使用用户明确给出的路径。
2. 如果用户只给出 `wiki/README.md`，则 `wikiDir` 是其父目录，`wikiIndex` 是该文件。
3. 如果用户只说“根据 wiki 生成 todo”，优先查找当前项目中已有的 wiki 索引，例如 `<wikiDir>/README.md`；不要假设只有根目录 `wiki/README.md`。
4. 如果用户只说“根据 docs 生成 wiki”，优先查找当前项目中明显的 Markdown 文档目录，例如 `docs/`；如果存在多个候选且无法判断，先询问用户。
5. 如果用户要求输出到 `todos.md`，按要求写入；否则将任务计划写入用户指定的 `todoOutput`，没有指定时才默认使用项目根目录的 `todos.md`。

在汇报中说明实际采用的路径。

## 生成 wiki 流程

参考 `docs-wiki-builder` 的方法，但保持目录通用：

1. 收集 `sourceDocsDir` 下的 Markdown 文件。
   - 默认只处理该目录内的 `.md` 文件。
   - 是否递归处理子目录由用户要求或项目结构决定；不确定时先说明假设。
2. 按文档标题拆分。
   - 解析标题时忽略代码块中的 `#`、`##`、`###`。
   - 不拆断表格、代码块、流程图、连续枚举步骤。
   - 尽量原样迁移正文，只添加必要导航。
3. 在 `wikiDir` 下生成：
   - 总目录 `wikiIndex`。
   - 每个源文档的主题入口。
   - 按二级/三级标题拆分的小文档。
4. 总目录包含：
   - 快速入口。
   - 推荐检索路径。
   - 关系图谱或关系表。
   - 细分文档树。
5. 每个小文档按需要添加简短导航：
   - 来源文档。
   - 总目录位置。
   - 上一篇/下一篇。
   - 相关主题。

关系和推荐路径必须从实际文档标题、编号、链接和语义中推断；不要创造源文档中不存在的主题。

## 生成 todo list 流程

当用户要求“根据 wiki/README.md 生成 vibe coding 任务计划，要求先简单后复杂；每个任务指向对应的 wiki 小文档；任务计划放到 todos.md 文档”时，执行以下流程。

1. 读取 `wikiIndex`。
   - 提取快速入口、推荐检索路径、关系图谱、细分文档树。
   - 以索引中真实存在的链接作为任务引用来源。
2. 必要时读取相关 wiki 小文档。
   - 优先读取任务所需的小文档，而不是一次性加载整个 wiki。
   - 任务标题和验收标准必须来自 wiki 内容或其明确关系，不要编造产品能力。
3. 按“先简单后复杂”组织任务。
   - L1：项目启动、基础契约、健康检查、最小数据结构。
   - L2：核心 CRUD 和页面/接口入口。
   - L3：工作流、状态机、日志、可追踪性。
   - L4：AI 执行、分析、拆解、沙箱或外部工具集成。
   - L5：Git、PR、自动化闭环。
   - L6：插件、扩展点、Skills、测试用例等扩展能力。
   - L7：监控、部署、多租户、测试策略、生产化能力。
   - 如果 wiki 主题不同，按实际主题调整层级；保持由低依赖到高依赖、由单模块到跨模块闭环。
4. 每个任务必须包含：
   - 任务编号，例如 `T01`、`T02`。
   - 目标：一句话说明要做什么。
   - 参考 wiki 小文档：至少一个真实存在的 wiki 链接。
   - 验收：可检查的完成标准。
5. 写入 `todoOutput`。
   - 如果目标文件已存在，先读取并判断是覆盖、更新还是合并；用户已明确要求“生成/放到”该文件时，可以更新该文件，但不要改无关内容。
   - 不额外创建 changelog、summary、临时检查报告。

推荐结构：

```md
# Vibe Coding 任务计划

来源：`<wikiIndex>`。

## 执行约定

- 每个任务先读参考 wiki 小文档。
- 每个任务只实现当前目标。
- 每个任务完成后按验收标准检查。

## L1 入门：...

- [ ] T01 ...
  - 目标：...
  - 参考 wiki 小文档：
    - [...](...)
  - 验收：...
```

## 校验要求

生成或更新 wiki 后检查：

- 源文档目录没有被修改。
- `wikiIndex` 能到达主题入口和小文档。
- 小文档相对链接能指向已生成文件。
- 代码块没有被标题解析误拆。

生成或更新 todo list 后检查：

- 任务层级是从简单到复杂。
- 每个任务都有目标、参考 wiki 小文档和验收标准。
- 每个任务至少有一个 wiki 链接。
- todo list 中所有 wiki 链接都能指向真实文件。
- 任务编号连续，不重复。

如果用户要求“循环检查 N 次”，用同一套确定性检查重复 N 次，并汇报每轮结果，例如：

```text
ROUND=1 LEVELS=... TASKS=... LINKS=... MISSING=...
ROUND=2 LEVELS=... TASKS=... LINKS=... MISSING=...
ROUND=3 LEVELS=... TASKS=... LINKS=... MISSING=...
```

## 汇报格式

完成后简短说明：

- 使用的源文档目录、wiki 目录、wiki 索引和 todo 输出文件。
- 创建或更新了哪些文件。
- 校验结果，尤其是链接缺失数量。
- 如果无法完成自动校验，明确说明原因和残余风险。
